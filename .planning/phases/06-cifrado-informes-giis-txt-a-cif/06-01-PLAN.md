# Plan 06-01 — Backend: Cifrado automático y eliminación del gate

**Objetivo:** Integrar la generación de CIF y ZIP en el flujo de finalización del batch (automático al completar). Eliminar el gate `GIIS_ENCRYPTION_VALIDATED`. Mantener endpoints de descarga TXT y ZIP; el endpoint `build-deliverable` deja de ser necesario como paso separado pero puede mantenerse como no-op o eliminarse.

**Contexto (06-CONTEXT.md):** Cifrado obligatorio y siempre activo. 1 ZIP por guía (CEX, LES). TXT + CIF + ZIP siempre generados. CDT fuera de alcance.

---

## must_haves

- [ ] **Cifrado automático al completar batch:** Cuando el batch pasa a status `completed` y `validationStatus !== 'has_blockers'`, ejecutar la lógica de cifrado (TXT → CIF → ZIP) para cada artifact. Si `validationStatus === 'has_warnings'`, ejecutar igual (sin exigir confirmación; el gate de confirmación se elimina).
- [ ] **Integración en recordBatchCompletionAudit o flujo de completion:** Llamar a la lógica cifrar+ZIP (extraída o reutilizada de buildDeliverable) justo antes o después de `recordBatchCompletionAudit`. Ubicación: en `generateBatchCex` cuando `isFirstGenerator` y se establece completed, y en `generateBatchLes` cuando se detecta que el batch quedó completed. Evitar duplicar: la lógica debe ejecutarse una sola vez cuando el batch tiene todos los artifacts (CEX y LES).
- [ ] **Eliminar gate GIIS_ENCRYPTION_VALIDATED:** Quitar toda comprobación de `GIIS_ENCRYPTION_VALIDATED` en el controller y en el servicio. El cifrado se ejecuta siempre si la clave `GIIS_3DES_KEY_BASE64` está configurada. Si la clave no está configurada, el batch falla con error claro (cifrado es obligatorio).
- [ ] **Endpoint build-deliverable:** Eliminar el endpoint POST `batches/:batchId/build-deliverable` o convertirlo en no-op que devuelva el batch actual (los ZIP ya existirán). Si se elimina, actualizar la documentación o tests que lo referencien.
- [ ] **Endpoints de descarga:** Mantener `GET batches/:batchId/download/:guide` (TXT) y `GET batches/:batchId/download-deliverable/:guide` (ZIP). Ambos deben estar disponibles cuando el batch está completed. El ZIP requiere que `artifact.zipPath` exista (generado automáticamente).
- [ ] **Mismo basename para TXT, CIF, ZIP:** Ya implementado en Phase 2. Verificar que los nombres sigan el patrón `[TIPO]-[ENTIDAD][INST]-[AA][MM].[EXT]`.
- [ ] **Tests:** Actualizar o crear tests que verifiquen: (1) al completar batch sin bloqueantes, los artifacts tienen `zipPath` y el ZIP existe; (2) no existe gate GIIS_ENCRYPTION_VALIDATED; (3) download y download-deliverable sirven los archivos correctos.

## Tasks

<task>
**Task 06-01.0 — Extraer lógica de cifrado a método privado**
- En `GiisBatchService`, extraer la lógica central de `buildDeliverable` (lectura TXT, cifrado, escritura CIF, creación ZIP, actualización artifacts con zipPath y hashSha256) a un método privado `encryptAndZipArtifacts(batchId: string): Promise<void>`.
- El método valida: batch existe, status completed o generating, validationStatus !== 'has_blockers'. Obtiene clave de GIIS_3DES_KEY_BASE64; si falta, lanza ConflictException. Itera artifacts, para cada TXT: encryptToCif, write CIF, createZipWithCif, write ZIP, actualiza artifact con zipPath y hashSha256.
- `buildDeliverable` (si se mantiene) delega en este método y devuelve el batch actualizado.
</task>

<task>
**Task 06-01.1 — Integrar cifrado en flujo de completion**
- En `generateBatchCex`: después del `updateOne` que establece status completed (cuando `isFirstGenerator`), si el batch tiene 2 artifacts (CEX + LES), llamar a `encryptAndZipArtifacts(batchId)`. Si solo tiene 1 (solo CEX porque LES falló o no se generó), igual llamar para los artifacts existentes.
- En `generateBatchLes`: después del `updateOne`, si `completedBatch?.status === 'completed'`, llamar a `encryptAndZipArtifacts(batchId)` antes de `recordBatchCompletionAudit`. Así evitamos duplicar: CEX como primer generador no llama a encrypt (porque en ese momento solo hay 1 artifact); LES al completar llama porque ya hay 2. Alternativa: centralizar en `recordBatchCompletionAudit` que llame a encryptAndZipArtifacts si los artifacts aún no tienen zipPath.
- Decisión: llamar `encryptAndZipArtifacts` solo cuando el batch está completed y al menos un artifact no tiene zipPath. Así se ejecuta una vez al final.
</task>

<task>
**Task 06-01.2 — Eliminar gate GIIS_ENCRYPTION_VALIDATED**
- En `GiisExportController`: eliminar la comprobación `process.env.GIIS_ENCRYPTION_VALIDATED?.toLowerCase() === 'true'` y el `throw new ConflictException(GIIS_ENCRYPTION_VALIDATED_MSG)` en el handler de `build-deliverable` (o eliminar el handler).
- Eliminar la constante `GIIS_ENCRYPTION_VALIDATED_MSG` si ya no se usa.
- En `GiisBatchService.buildDeliverable` (si se mantiene): no hay gate; la clave es el único requisito.
</task>

<task>
**Task 06-01.3 — Endpoint build-deliverable y mensaje download-deliverable**
- Opción A: Eliminar POST `batches/:batchId/build-deliverable` del controller. Actualizar tests que lo invoquen.
- Opción B: Mantener el endpoint pero hacer que delegue en `encryptAndZipArtifacts` si los artifacts no tienen zipPath; si ya tienen, devolver el batch sin cambios. Útil para reintentos manuales.
- Recomendación: Opción B para compatibilidad con clientes que puedan llamarlo; el flujo normal no lo requiere.
- **Actualizar mensaje de error en download-deliverable:** Si `!artifact.zipPath`, cambiar el mensaje de "Ejecute build-deliverable primero" a uno neutro: "No existe entregable ZIP para la guía {guide}." (evitar referencias al paso build-deliverable obsoleto).
</task>

<task>
**Task 06-01.4 — Tests**
- Crear o extender `test/nom024/giis-batch.nom024.spec.ts`: (1) Al completar batch (CEX+LES) sin bloqueantes, cada artifact tiene zipPath y el archivo ZIP existe en disco. (2) download-deliverable devuelve el ZIP con Content-Disposition correcto. (3) Con GIIS_ENCRYPTION_VALIDATED=false (o variable no definida), el cifrado automático se ejecuta igual (no hay 409). (4) Si GIIS_3DES_KEY_BASE64 no está configurada, el completion falla con error apropiado.
</task>
